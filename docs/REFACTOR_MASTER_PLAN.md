# REFACTOR_MASTER_PLAN.md

## 1. Executive Summary
The objective is to transition from a prototype-grade Vanilla JS implementation to a **Declarative, Component-Based Architecture** using the "Golden Stack" of WebGL.

**Key Architectural Shifts:**
1.  **Rendering:** From imperative `new Mesh()` calls to declarative **React Three Fiber (R3F)** components.
2.  **Assets:** From hardcoded JS voxel arrays to an external **GLB Asset Pipeline** (MagicaVoxel → GLB → React Component).
3.  **Performance:** From individual draw calls to **Hardware Instancing** for static geometry.
4.  **Logic:** From array iteration to **O(1) Hash Maps** for recipe validation.

## 4. Phase 1: The Asset Pipeline (The "Golden Standard")
**Goal:** Decouple code from art. remove `VoxelBuilder.js`.

### 4.1 The Workflow
1.  **Creation:** Artist creates assets in **MagicaVoxel**.
2.  **Export:** Export to **GLB**.
3.  **Optimization & Code Gen:** Developer runs `npx gltfjsx public/models/item.glb --transform --types`.
4.  **Result:** A TypeScript React Component (`src/components/Models/Item.tsx`) that loads the compressed asset from the CDN/Public folder.

### 4.2 Component Strategy
*   **Dynamic Items (Food, Tools):** Use the generated React Components directly. They allow for independent movement, animation, and parent changing.
*   **Static World (Walls, Floors):** Do **not** render 1000 `<Wall />` components.
    *   Extract the geometry from the GLB hook: `const { nodes } = useGLTF('/wall.glb')`.
    *   Feed this geometry into `<Instances>` from `@react-three/drei`.

## 5. Phase 2: Domain Modeling (TypeScript)

### 5.1 Interfaces (`/src/types/GameTypes.ts`)
*   **`ItemType`**: Union of keys mapping to GLB files (e.g., `'burger_bun' | 'tomato'`).
*   **`EntityId`**: String (UUID).
*   **`LevelSchema`**: Strict typing for JSON level files.
*   **`RecipeHash`**: Number (Prime Product).

---

## 6. Phase 3: The Rendering Core

### 6.1 Level Geometry (`/components/World/LevelGeometry.tsx`)
This component receives the JSON layout.
*   Filters layout for static types (Walls, Floor, Counters).
*   Uses `useMemo` to calculate transformation matrices.
*   Renders `<InstancedMesh>` nodes.
*   **Benefit:** 1 Draw call per asset type, regardless of level size.

### 6.2 Dynamic Entities
*   Rendered as standard React components based on the `useGameStore` entity list.
*   Managed by R3F for automatic disposal (preventing memory leaks).

---

## 7. Phase 4: Logic Systems (The "Brain")

### 7.1 State Management (`/store/useGameStore.ts`)
Split state to prevent React performance bottlenecks.
*   **Reactive (Zustand):** Score, Current Level, Inventory UI. (Triggers DOM updates).
*   **Transient (Refs):** Player Position, Camera Rotation. (Accessed directly in `useFrame` loop).

### 7.2 The Recipe System (Optimization: O(1))
Replace array iteration with **Prime Number Hashing**.
*   **Concept:** Every ingredient has a unique prime number ID.
*   **Calculation:** Plate Content Value = Product of Ingredient Primes.
    *   *Example:* Bun(2) * Patty(3) * Cheese(5) = 30.
*   **Validation:** `RECIPES[30] = 'Cheeseburger'`.
*   **Benefit:** Constant time lookup. No sorting arrays. Instant validation.

### 7.3 Interaction System (Strategy Pattern)
*   **Strategies:** `PickupStrategy`, `PlaceStrategy`, `CombineStrategy`, `ProcessStrategy`.
*   **Logic:**
    ```typescript
    // Inside useInteraction hook
    const handler = strategies.find(s => s.canExecute(heldItem, target));
    handler?.execute();
    ```

---

## 8. Phase 5: The Player & Controls

### 8.1 Player Controller
*   Use `@react-three/drei` `KeyboardControls` for input mapping.
*   **Physics:** Simple AABB collision against the Grid Map (calculated from Level JSON).
*   **Interaction Raycaster:**
    *   Only active when `GameState === 'PLAYING'`.
    *   Throttled: Run logic every frame, but only update React State if the hit target *changes* (deduplication).

---

## 9. Performance Mandates (Strict)

1.  **Assets:** All GLBs must be processed with `--transform` (Draco Compression).
2.  **Instancing:** Any static object appearing >3 times must be Instanced.
3.  **Loops:** No `new` keyword allocations inside `useFrame` (Vector3, Quaternion, etc., must be reused).
4.  **Re-renders:** The HUD must pull specific state slices (e.g., `useGameStore(state => state.score)`), never the whole state object.
